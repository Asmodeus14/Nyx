Phase 1: PCI Initialization & Memory Mapping (The Foundation)
Before you can draw anything, the kernel must find the GPU and map its memory into your virtual address space.

Reference: vol13-mmio.pdf

Actionable Steps:

PCI Enumeration: Scan the PCI bus for Vendor ID 0x8086. Match the Device ID against known Kaby Lake / Comet Lake IDs (e.g., 0x5916, 0x9BC5, etc.).

Map BAR0 (GTTMMADR): This is the most critical step. BAR0 contains the Graphics Translation Table (GTT) and all the Memory Mapped I/O (MMIO) registers. Read this physical address from the PCI configuration space and map it into your kernel's virtual memory as Uncacheable (UC) to prevent the CPU from caching register writes.

Map BAR2 (Aperture): Map the aperture base. This is a CPU-visible window into the GPU's memory.

Wake the Device: Ensure the device is powered up by checking the PCI Power Management registers and writing to the appropriate MMIO power control registers to wake the graphics engines.

Phase 2: The Global Graphics Translation Table (GGTT)
The GPU does not understand your CPU's page tables; it has its own memory management unit (IOMMU/GART).

Reference: vol06-command_stream_programming.pdf (Memory Interface section) & vol13-mmio.pdf.

Actionable Steps:

Understand the GGTT: The GGTT maps GPU virtual addresses to physical memory addresses. The PTEs (Page Table Entries) for the GGTT are located exactly at GTTMMADR + 2MB.

Allocate Backing Pages: Using nyx-kernel/src/allocator.rs, allocate physical pages in system RAM to serve as your framebuffers and command buffers.

Write PTEs: Write the physical addresses of these allocated pages into the GGTT. Now, when the GPU tries to access a specific graphics virtual address, it will fetch from your allocated physical system RAM.

Phase 3: Ring Buffers and Command Streamers (Talking to the GPU)
You don't control the GPU by constantly poking registers. You write commands to a circular queue in memory (a Ring Buffer), and the GPU's Command Streamer reads and executes them asynchronously.

Reference: vol06-command_stream_programming.pdf & vol02a-commandreference-instructions.pdf.

Actionable Steps:

Setup the BCS (Blitter Command Streamer): For UI offloading, the Blitter engine is your best friend. Allocate a page for the Blitter Ring Buffer. Map it via the GGTT.

Initialize Ring Registers: Write the starting address, length, and enable bit to the Ring Buffer MMIO registers (BLT_RING_BASE, BLT_RING_CTL).

Write the first Command: Write a MI_NOOP (No Operation) to the ring buffer.

Ring the Doorbell: Update the Ring Buffer Tail pointer register. The GPU will see the tail has moved, fetch the MI_NOOP, execute it, and update the Head pointer.

Phase 4: Hardware-Accelerated 2D UI (The Blitter Engine)
This is where your software loops in libs/gui/src/draw.rs get replaced by hardware commands. We will use the GPU to composite windows and fill rectangles.

Reference: vol11-blitter.pdf & vol02a-commandreference-instructions.pdf.

Actionable Steps:

XY_COLOR_BLT (Rectangle Fill): Instead of looping over pixels to draw a window background, construct an XY_COLOR_BLT command packet. You specify the X/Y coordinates, width, height, destination memory pitch, and color. Push this to the Blitter Ring Buffer and advance the tail.

XY_SRC_COPY_BLT (Bitmask / Window Compositing): When your compositor needs to move a window's backing buffer to the main screen, use this command. It copies pixel blocks from a source address to a destination address asynchronously at massive speeds.

Syncing: Implement a way to wait for the GPU to finish. You can use MI_STORE_DATA_IMM to write a sequence number to a specific memory address when a batch of commands finishes, and have the CPU spinloop (or sleep/interrupt) waiting for that value to change.

Phase 5: Display Engine and Modesetting (Native Resolution)
Currently, you likely inherit a framebuffer from your bootloader (EFI GOP). Taking native control allows you to change resolutions, setup double-buffering, and use hardware cursors.

Reference: vol12-display.pdf.

Actionable Steps:

The Display Pipeline: Understand the flow: Planes (memory buffers) -> Pipes (blenders/color formatters) -> Transcoders (timing generators) -> Ports/DDI (physical connectors like HDMI/eDP).

Configure a Plane: Point a primary display plane to your GGTT-mapped framebuffer.

Hardware Cursor: Intel GPUs have dedicated registers for a hardware mouse cursor plane. This means moving the mouse requires zero redrawing of the UI underneath it. Hook this up to nyx-kernel/src/mouse.rs.

Phase 6: The 3D Render Command Streamer (RCS)
Once the 2D Blitter is handling your UI compositing, you can tackle the beast: the 3D pipeline.

Reference: vol07-3d_media_gpgpu.pdf.

Actionable Steps:

Initialize the RCS: Setup a ring buffer for the Render Command Streamer, just like you did for the Blitter.

Pipeline State: The Gen9.5 3D pipeline requires setting up dozens of states (Vertex Fetch, Geometry Shader, Rasterizer, Pixel Shader). You will send 3DSTATE_* commands to configure the Unified Return Buffer (URB) and pipeline geometry.

Shaders: You will eventually need to write or compile GEN assembly instructions (EU ISA) to tell the Execution Units (EUs) how to shade pixels.

   - Allocate command buffer

   - Enable command streamer (0x020C)

Implementation Notes:
====================

1. Use existing pci.rs for all hardware detection - no external tools needed

2. All testing is on bare metal - no QEMU emulation

3. Reference Intel CML PRM (Doc Ref # IHD-OS-CML-Vol 1-4.20) for:
   - Register definitions
   - Initialization sequences
   - Device attributes
   - Stepping and Device IDs

4. Key Comet Lake architectural details:
   - GT1F: 1x2x6 (1 Slice, 2 Subslices, 6 EUs per subslice = 12 EUs)
   - GT2: 1x3x8 (1 Slice, 3 Subslices, 8 EUs per subslice = 24 EUs)
   - 48-bit virtual address range
   - 39-bit physical address range
   - 3 display pipes, 3 planes per pipe
   - 2 DDI ports + 1 eDP port

5. Critical registers to implement:
   - GFX_MODE (0x915C) - Graphics mode control
   - PIPE_CONF (0x70008) - Pipe configuration
   - PIPE_SRC (0x6001C) - Pipe source size
   - GTT_BASE (0x0188) - GTT base address
   - GTT_CTRL (0x0180) - GTT control
   - RCS_BASE (0x0200) - Render command streamer base
   - RCS_CTRL (0x020C) - Render command streamer control

6. Testing approach for bare metal:
   - Serial port for debug output
   - Motherboard LEDs/GPIO for status
   - Memory validation routines
   - Thermal monitoring
   - Fallback to CPU rendering if needed

7. Performance optimization priorities:
   - Command batching to reduce syscall overhead
   - Memory pooling for frequently used resources
   - Double buffering for smooth animation
   - Partial updates for damaged regions
   - Hardware cursor for pointer rendering
