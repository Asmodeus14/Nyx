Milestone 1.1: Fix create_file Stub ✅ Priority

Problem: fs.rs line 147: fn create_file(&mut self, _path: &str) -> bool { true } returns success without touching the disk. Any code path relying on explicit file creation (like the Phase 5 installer pre-allocating a placeholder) will silently fail.
Action:
In ext4_wrapper.c, add:
c

int nyx_fs_create_file(const char* path) {
    ext4_file f;
    if (ext4_fopen(&f, path, "w+") != EOK) return 0;
    ext4_fclose(&f);
    return 1;
}
In fs.rs, declare fn nyx_fs_create_file(path: *const u8) -> i32; in the extern block and implement:
rust

fn create_file(&mut self, path: &str) -> bool {
    let c_path = to_c_path(path);
    unsafe { nyx_fs_create_file(c_path.as_ptr()) == 1 }
}
Result: Explicit file creation works correctly and returns actual success/failure.
Milestone 1.2: GPT Partition Selection by Type GUID ✅ Priority

Problem: fs.rs line 106 uses a hardcoded 90-110 GB size window: (size_mb_512 > 90_000 && size_mb_512 < 110_000) || (size_mb_4k > 90_000 && size_mb_4k < 110_000). This is testing-specific and will break when you move to full disk or a different partition size.
Action: Replace the size check with a type GUID match. The standard Linux filesystem GPT type GUID is 0FC63DAF-8483-4772-8E79-3D69D8477DE4 (stored little-endian on disk). In the GPT entry parsing loop:
rust

// Read the full 16-byte type GUID (bytes 0-15 of entry)
let mut type_guid = [0u8; 16];
type_guid.copy_from_slice(&entry_block[off..off+16]);

// Linux filesystem data partition GUID (little-endian on disk)
const LINUX_FS_GUID: [u8; 16] = [
    0xAF, 0x3D, 0xC6, 0x0F, 0x83, 0x84, 0x72, 0x47,
    0x8E, 0x79, 0x3D, 0x69, 0xD8, 0x47, 0x7D, 0xE4
];

if type_guid == LINUX_FS_GUID {
    start_lba = lba;
    size_sectors = end_lba - lba;
    break; // Take the first Linux partition
}
Result: Nyx mounts any Linux-formatted GPT partition regardless of size. The 100 GB test partition still works; full-disk installs will work too.
Milestone 1.3: Add delete_file to the VFS Trait (Suggestion)

Problem: The FileSystem trait in vfs.rs has no delete_file method. The Phase 5 git micro-agent will need to delete stale firmware caches, and the app store will need to uninstall apps.
Action: Add fn delete_file(&mut self, path: &str) -> bool { false } as a default method (returning false means "unsupported"). Implement it in NvmeLwExt4Fs by adding nyx_fs_remove in ext4_wrapper.c calling ext4_fremove.
Result: File deletion is available to userspace when needed.
Milestone 1.4: Increase Block Cache Size (Suggestion)

Problem: build.rs line 65 sets CONFIG_BLOCK_DEV_CACHE_SIZE 16. lwext4 caches 16 blocks (8 KB at 512-byte sectors) in RAM. For sequential reads (like loading an app binary), this causes excessive NVMe round-trips because the cache thrashes constantly.
Action: Increase to 128 or 256 (64-128 KB cache). This costs negligible RAM (a few hundred KB) but dramatically reduces NVMe command overhead.
Result: Faster file reads, especially for the initrd tarball extraction and app loading.
Milestone 1.5: Distinguish Error Types in Return Values (Suggestion)

Problem: read_file returns Option<usize> — None could mean "file not found" or "disk I/O error." Userspace (and the kernel itself) can't distinguish recoverable from fatal errors.
Action: Change Option<usize> to Result<usize, FsError> where FsError is an enum (NotFound, IoError, InvalidPath, OutOfSpace). This is a breaking change to the trait but worth doing now before more callers exist.
Result: Proper error reporting to userspace via errno-style syscalls.
Milestone 1.6: Verify 4K Native NVMe Sector Handling (Suggestion)

Problem: fs.rs allocates an 8192-byte buffer, manually aligns it to 4096, then calls driver.read_block(sector, slice_4k) and copies only 512 bytes out. The comment says "We MUST pass 4096 to bypass the hardcoded safety check in nvme.rs!". This suggests the NVMe driver expects 4K buffers but lwext4 works in 512-byte logical sectors. If the underlying NVMe device is 4K-native (no 512-byte emulation), this works. But if it's 512-byte emulation, you're doing 8x more I/O than necessary.
Action: Verify your NVMe driver's logical block size. If it's 512, simplify the bridge to pass 512-byte buffers directly. If it's 4K-native, keep the current approach but document why. The block size mismatch is a common source of silent data corruption.
Result: Correct and efficient block I/O.
Milestone 1.7: Add sync/flush Operation (Suggestion)

Problem: lwext4 has ext4_cache_flush() and ext4_fs_sync() to force journal commits. The FileSystem trait has no sync method. In a power-loss scenario, unflushed journal data could corrupt the filesystem.
Action: Add fn sync(&mut self) -> bool { true } as a default method. Implement it in NvmeLwExt4Fs by calling ext4_cache_flush and ext4_fs_sync in the C wrapper.
Result: Clean shutdown and power-loss safety.
Phase 2: Ethernet Stack Hardening (Primary Focus)
This is where you spend the next 4-6 weeks. The goal is to transform the RTL8168 driver from a Windows-ICS testbench prototype into a robust, interrupt-driven, DHCP-capable stack that survives real network traffic.

Milestone 2.1: Dynamic IP via DHCP ✅ Priority

Problem: pci.rs hardcodes 192.168.137.99/24 with gateway 192.168.137.1.
Action: Enable smoltcp's dhcp feature. Replace the static IpCidr assignment with a Dhcpv4Client instance stored alongside NET_IFACE. In poll_network(), call dhcp_client.poll() before iface.poll().
Result: Nyx auto-configures on any Ethernet network.
Milestone 2.2: Interrupt-Driven RX ✅ Priority

Problem: pci.rs configures MSI vector 0x30 but no handler exists. poll_network() spins on every tick.
Action: Register an ISR for vector 0x30 that sets NETWORK_PENDING.store(true, Relaxed) and issues an EOI. Rewrite poll_network() to idle (HLT) until NETWORK_PENDING is set, then poll. Remove the write16(0x3E, 0xFFFF) from receive().
Result: CPU usage drops to ~0% when idle; no dropped packets under burst traffic.
Milestone 2.3: Hot-Path Debug Gating ✅ Priority

Problem: serial_println! on every RX/TX packet will stall the kernel under real traffic.
Action: Add a net_trace feature in nyx-kernel/Cargo.toml. Gate all packet-level serial prints behind #[cfg(feature = "net_trace")].
Result: Zero serial overhead in production builds.
Milestone 2.4: RX Buffer Pool ✅ Priority

Problem: receive() does alloc::vec![0; length] on every packet, fragmenting the heap.
Action: Pre-allocate a pool of 16 Box<[u8; 2048]> in Rtl8168Driver::new(). receive() checks out a buffer from the pool, copies the frame, and returns ownership. RtkRxToken returns the buffer to the pool on Drop.
Result: Zero per-packet heap allocation after boot.
Milestone 2.5: PCI ID Coverage & TSC Calibration

Problem: Only 0x10ec:0x8168 binds. TSC assumed 2 GHz.
Action: Widen the match to matches!(device_id, 0x8168 | 0x8125 | 0x8169 | 0x8136 | 0x8167). Read the actual TSC frequency from ACPI or calibrate against the HPET/local APIC timer.
Result: Driver binds to full RTL81xx family; TCP timers run correctly on any CPU.
